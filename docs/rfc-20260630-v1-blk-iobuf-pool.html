blk_iobuf_pool
per-queue kernel I/O buffers for LBS storage

A per-request_queue pool of higher-order folios sized to the device's geometry, plus an io_uring opcode that hands user rings kernel-allocated, device-aligned fixed buffers — cutting per-I/O CPU and NVMe descriptor pressure on Large-Block-Size and segment-constrained devices.

01The problem

Storage is moving to Large Block Size (LBS) media and to controllers with a large transfer size but a limited segment budget. On both, the device's natural unit of transfer is no longer the 4 KiB CPU page, and the buffers the kernel is handed for I/O are almost never shaped the way the device wants them.

An application doing high-IOPS O_DIRECT hands the kernel a virtual buffer. Physically that buffer is a scatter of 4 KiB pages (or, at best, a transparent huge page that happens to be 2 MiB — not pinned to the device block). Two costs follow, on every I/O:

⊟Segment explosion

A 16 KiB I/O from 4 KiB pages is up to 4 bio segments; a 64 KiB I/O is up to 16. Each segment becomes an NVMe PRP/SGL descriptor. On a device whose max_segments is small relative to its transfer size, fragmented buffers force extra requests, or fail the segment budget outright.

↻Per-I/O import cost

io_uring registered ("fixed") buffers from userspace still pay the full io_import_fixed() path on every submission: overflow/bounds validation, direction checks, folio-mask and segment-offset arithmetic. That fixed cost is pure overhead for a buffer the kernel could have trusted by construction.

There is no kernel facility that owns a supply of correctly-sized, physically-contiguous, geometry-aligned buffers per device. Every consumer — io_uring fixed buffers, filesystem direct I/O, NVMe passthrough — re-solves (or fails to solve) the same alignment problem on its own.

02Workload use cases

①High-IOPS io_uring

Databases & storage engines (RocksDB, ScyllaDB, SPDK-style designs) doing O_DIRECT with registered buffers at device granularity, at deep queue depths across many cores. The hot path is per-I/O software cost — exactly what kernel-backed buffers shrink.

②Filesystem direct I/O

XFS / ext4 on a 16 KiB-LBS device: O_DIRECT writes must be block-aligned, and a contiguous, device-aligned buffer lets iomap submit a single-segment bio instead of a fragmented one.

③NVMe passthrough

ioctl passthrough and other in-kernel producers that need a bounce buffer matching device geometry, with no userspace API to lean on at all.

scope All three share one missing primitive: a per-queue supply of geometry-sized folios. Provide it once in the block layer and every consumer benefits.

03When the pool activates — device geometry

A pool is created at queue-limits-commit time by blk_iobuf_choose_order(), which inspects three device-geometry signals. A queue gets a pool only if at least one fires; the folio order is the largest size implied, clamped to [PAGE_SIZE, PAGE_SIZE << iobuf_pool_max_order]. The signals that fired are recorded as a bitmask in /sys/block/<dev>/queue/iobuf_pool_reasons.

◆IO_MIN0x1

io_min > PAGE_SIZE — the device reports a minimum I/O size larger than the CPU page.

When: a Large Block Size NVMe namespace (16 KiB logical block → io_min 16 KiB), or a drive advertising a 16 KiB write/indirection granularity (NPWG).

Example: a 16 KiB-LBS or 16 KiB-IU drive → order 2 (16 KiB folios).

◆SEG_GEOM0x2

max_hw_bytes / max_segments > PAGE_SIZE — the device moves a lot per command but accepts relatively few scatter-gather segments, so each segment must be large.

When: a controller with a big MDTS (max transfer) but a modest max_segments.

Example: an 8 MiB MDTS drive with 128 segments → 8 MiB/128 = 64 KiB/segment → fires even if the logical block is 512 B/4 KiB → order 4 (64 KiB). (With a very large max_segments the ratio can fall to ≤ PAGE_SIZE and this will not fire — read the bitmask to know.)

◆IO_OPT0x4

io_opt >= PAGE_SIZE and the admin opted in (iobuf_pool_prefer_io_opt=1) — the device advertises an optimal (advisory) I/O size.

When: RAID/stripe arrays or SSDs reporting an optimal write size you want to size the pool to.

Example: a 64 KiB-stripe array → order 4 once the opt-in is set. Off by default because io_opt is a hint, not a requirement.

 read back what fired, on any device
cat /sys/block/nvme0n1/queue/iobuf_pool_reasons    # bitmask: 0x1 IO_MIN | 0x2 SEG_GEOM | 0x4 IO_OPT
cat /sys/block/nvme0n1/queue/iobuf_pool_order      # resulting folio order (size = PAGE_SIZE << order)
cat /sys/block/nvme0n1/queue/iobuf_pool_folio_size # resulting folio size in bytes
# the geometry that drives the decision:
cat /sys/block/nvme0n1/queue/{minimum_io_size,optimal_io_size,max_hw_sectors_kb,max_segments}
note A queue where all three signals stay at or below PAGE_SIZE gets order 0 and no pool is created — the feature is inert on hardware that does not need it.

04Today's options — and why they fall short

ApproachWhat it doesWhy it isn't enough
App-side aligned alloc
posix_memalign / hugetlb		 Each application allocates aligned, contiguous memory itself. Burden duplicated into every app; fragile (no guarantee of physical contiguity below a huge page); wastes memory (2 MiB hugepage for a 16 KiB need); does nothing for in-kernel consumers (passthrough, iomap).
io_uring user fixed buffers
IORING_REGISTER_BUFFERS		 Pre-registers and pins user pages so per-I/O page lookup is cached. Still runs the heavy io_import_fixed() validation every I/O, and inherits whatever physical fragmentation the user pages have. The buffer is only as aligned as userspace made it.
Transparent huge pages
THP / MADV_HUGEPAGE		 Opportunistically backs the buffer with 2 MiB pages, improving contiguity. Best-effort and allocation-policy dependent; granularity (2 MiB) is mismatched to the block (16 KiB); not pinned to geometry; invisible to the kernel as "a block-sized buffer."

In short: the alignment problem is pushed to userspace, solved partially and repeatedly, and never made available to kernel I/O producers.

05What this series adds

A single block-layer primitive — blk_iobuf_pool — and three consumers that use it.

◆The pool

A per-request_queue mempool of compound folios. The order is chosen from device geometry (§03) — order 2 (16 KiB) on a 16K-LBS NVMe. Built on a new mm folio-typed mempool helper. State and counters are exposed via queue sysfs, and five tracepoints instrument pool create / alloc / free / fallback / bounce.

①io_uring opcode 38

IORING_REGISTER_BUFFERS_ALLOC_FOR_FILE registers kernel-allocated pool folios as a ring's fixed buffers (IO_REGBUF_F_KBUF, ubuf=0). Because they are trusted kernel buffers, I/O dispatches to the lightweight io_import_kbuf() path instead of io_import_fixed().

②iomap DIO write bounce

iomap_dio_write_iobuf(): for a block-aligned, folio-sized O_DIRECT write, copy into one pool folio and submit a single-segment bio. Strictly gated — single folio only, never an atomic, bvec, or integrity (PI) write.

③NVMe passthrough + validation

Passthrough bios bounce through pool folios; the namespace creates its pool on limits commit, and the io_uring path validates the buffer's queue association so a buffer can't be used against the wrong device.

Two source-attributable wins

1 · Cheaper import. Kernel buffers skip the per-I/O range/direction validation and segment-offset arithmetic of io_import_fixed() — a roughly constant saving per I/O.

2 · Fewer segments / perfect alignment. One contiguous 16 KiB folio = one bio segment, versus up to four for 4 KiB pages → fewer NVMe PRP/SGL entries. The win grows on devices whose max_segments is small relative to their transfer size.

06Tuning — module parameters

Three parameters on the built-in blk_iobuf code govern pool sizing. They are boot parameters (blk_iobuf.<name>=…) and are also writable at runtime under /sys/module/blk_iobuf/parameters/ (mode 0644).

ParameterDefaultWhat it does
iobuf_pool_max_order 5
128 KiB		 Upper bound on the chosen folio order, so geometry can never inflate the pool past PAGE_SIZE << 5. Set to 0 to disable the pool entirely — the order clamps to PAGE_SIZE, choose_order() returns 0, and no pool is created. This is the knob the validation uses for its pool-off baseline.
iobuf_pool_min_folios 16 Minimum number of folios kept preallocated in each pool's mempool reserve, so the hot path has guaranteed forward progress under memory pressure before it falls back to page allocation.
iobuf_pool_prefer_io_opt off Opt in to letting io_opt raise the folio order (enables the BLK_IOBUF_REASON_IO_OPT signal of §03). Off by default because io_opt is advisory; enable it when you want the pool sized to a RAID stripe or an optimal write size. 
 examples
# disable the pool globally (pool-off baseline)
blk_iobuf.iobuf_pool_max_order=0          # boot param

# opt into io_opt sizing at runtime, then re-commit limits (re-scan / reload)
echo 1 | sudo tee /sys/module/blk_iobuf/parameters/iobuf_pool_prefer_io_opt

# cap the order at 4 (64 KiB) instead of 5 (128 KiB)
blk_iobuf.iobuf_pool_max_order=4

The pool is recomputed at queue-limits-commit time, so a runtime parameter change takes effect the next time a device commits limits (e.g. on rescan or driver reload), not retroactively on live queues.

08Reproduce with kdevops

The whole rig is a kdevops defconfig: it provisions a libvirt guest with an emulated 16 KiB-LBS NVMe and builds the branch straight from git.kernel.org. No LBS hardware required.

The iobuf-nvme defconfig pins both the test geometry and the kernel source — it now points bootlinux directly at this RFC's branch via the custom-tree mechanism:

 defconfigs/iobuf-nvme
CONFIG_LIBVIRT_EXTRA_STORAGE_NVME_LOGICAL_BLOCK_SIZE_16K=y   # 16K-LBS NVMe
CONFIG_BOOTLINUX_CUSTOM=y
CONFIG_BOOTLINUX_TREE_CUSTOM_URL="https://git.kernel.org/pub/scm/linux/kernel/git/mcgrof/linux.git"
CONFIG_BOOTLINUX_TREE_CUSTOM_REF="blk-iobuf-pool-v2"
required CONFIG_BLK_IOBUF_POOL is default n (EXPERIMENTAL). The built kernel must set CONFIG_BLK_IOBUF_POOL=y (and, for the LBS guest, large-block support) or the pool never exists.

4 · Real workload — LMCache KV-cache offload (kvio)

A concrete consumer study: does an LLM KV-cache-offload workload actually engage the pool? kvio drives LMCache's raw_block io_uring_cmd NVMe passthrough from real model geometry (fake, content-free KV bytes — storage geometry is independent of tensor values). Finding: that path submits IORING_OP_URING_CMD with page-aligned user buffers that map zero-copy, so iobuf_pool_allocs stays 0 under nvme_core.iobuf_pool off/auto/force alike — confirmed by a live ftrace (1028 URING submits, zero op-38 registers). The pool is fed only by opcode 38. A small opt-in (iobuf_pool_bench) adds a matched A/B on the same block-device ReadFixed engine:

 guest
# arm A: ordinary user buffer (op 0)   arm B: op-38 pool folios — same loop, same device
python examples/kv_cache_offload_io/iobuf_pool_bench.py --blkdev /dev/nvme1n1 --buffer-size 131072 --nr-ops 128

Result (16 KiB-LBS, order 2): arm A iobuf_pool_allocs +0, arm B +8 (128 KiB ÷ 16 KiB folios) — the pool engages only via op 38, never via kvio's passthrough path. Full test setup, findings and the A/B table: scripts/workflows/iobuf_bench/kvio-results.md. How kvio works, its dependencies and how to run it: kvio documentation page.

11Future work

The io_uring opcode is a ready, measured win and is the centerpiece of this RFC. The remaining items below are the deliberately-deferred edges — each is scoped, and most are already pointed at in the relevant commit messages.

Make the iomap bounce a win, not just regression-free

The shipped bounce is gated to a single folio and is regression-free, but on a device that is not segment-bound the extra memcpy only breaks even. The general win for larger writes needs a get_user_pages probe of the real, post-coalescing segment count: build the would-be bio, and bounce only when its segment count exceeds the queue budget — so a large contiguous (THP-backed) buffer, which the cheap iov_iter_npages estimate cannot distinguish from a fragmented one, stays on the zero-copy path. This is exactly where the multi-folio bounce (prototyped and shelved) earns its keep, and the natural test target is a segment-bound device — large MDTS, small max_segments (the SEG_GEOM case of §03), the hardware that most motivates the pool.

iomap PI / fscrypt parity

The bounce currently skips integrity (PI) writes — it does not yet replicate the normal path's fs_bio_integrity_generate(), so an IOMAP_F_INTEGRITY write is guarded out and takes the normal path. To bounce PI writes it must generate the integrity metadata on the bounced bio and be audited against fscrypt edge cases.

REQ_ATOMIC bounce

A single contiguous folio is, if anything, friendlier to hardware atomics than fragmented user pages, so a single-folio==full-length atomic write is a sound candidate to bounce. It needs full bio-setup parity (sector, integrity, write-hint, ioprio) and an RWF_ATOMIC + power-fail + fault-injection test matrix before it can be trusted; it is guarded out for now.

io_opt-driven larger orders, and wider consumers

With iobuf_pool_prefer_io_opt=1 a device advertising a large optimal I/O size raises the pool order and extends the single-segment sweet spot to bigger transfers — not yet exercised on real hardware. The pool primitive itself is consumer-agnostic; other bounce-needing paths (md/dm, additional filesystem DIO) could draw from it.

open question Whether the iomap bounce belongs in the first upstream posting at all, or should land later once the get_user_pages gate makes it a measured win, is a live review question. The io_uring opcode stands on its own and need not wait for it.
